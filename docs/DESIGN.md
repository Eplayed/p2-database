# PoE2 数据控制台设计规范

最后更新：2026-06-18

## 1. 设计目标

Dashboard 是个人开发者的数据运维台。它的价值是快速判断“该跑什么、跑到哪、结果是否可发布”，不是展示型网站。

优先级：状态清楚 > 操作安全 > 日志可读 > 视觉装饰。

## 2. 信息架构

首页只展示当前仍在维护的一键任务：

1. 日常数据更新并上传。
2. 天梯/BD解析并上传。

底层单项脚本默认隐藏，仅作为组合任务步骤。已经下架、低频或历史任务不重新放回主界面。

每个任务必须显示：

- 任务名称和用途。
- 环境 `release/dev`。
- 当前状态、上次运行时间和耗时。
- 执行步骤。
- 运行、停止操作。

## 3. 视觉基线

| 语义 | 推荐颜色 |
|---|---|
| 页面背景 | `#151515` |
| 面板背景 | `rgba(25, 25, 25, 0.92)` |
| 主文字 | `#eeeeee` |
| 次文字 | `#bcb6a8` |
| 品牌/进行中 | `#d4af37`、`#f0c86a` |
| 成功 | `#72c97c` |
| 失败/停止 | `#ff6b6b` |

- 面板圆角不超过 `8px`。
- 控制台采用紧凑信息密度，不使用大面积 Hero 或装饰卡片。
- 按钮尺寸固定，状态文字变化不能引发布局跳动。
- 不使用紫色渐变、装饰光球或无意义动画。

## 4. 控件规则

- 环境切换使用分段控件。
- 自动运行使用开关或明确状态按钮。
- 数值间隔使用输入框/步进器，不用文字按钮模拟。
- 运行、停止、刷新使用熟悉图标加文字。
- 危险操作必须二次确认，并明确说明影响的 OSS 环境。
- 模态框可通过按钮和 Escape 关闭，不能形成不可点击遮罩。

## 5. 日志体验

- 默认跟随最新日志。
- 用户主动向上滚动后立即暂停自动跟随。
- 显示“回到底部”按钮恢复跟随。
- stdout、警告和错误使用不同颜色，但保持可复制文本。
- 长任务必须能停止；停止后显示最终退出状态。
- 不在日志中输出 OSS 密钥、Cookie 或完整身份凭证。

## 6. 数据摘要

- 摘要只展示当前小程序仍消费的数据。
- `0` 与“文件不存在”必须区分。
- 显示更新时间和条数，避免只显示“成功”。
- release 数据异常时使用红色明确阻止上传，不以普通提示掩盖。
- 已下架的 starter/热门社区 BD 不再出现在摘要或任务中。

## 7. 验证清单

- `/api/tasks` 只返回当前可见任务。
- 任务执行步骤与界面描述一致。
- 手动滚动日志不会被强制拉到底部。
- 运行、停止、取消倒计时都可操作。
- 1280px 和 1440px 宽度无重叠，窄屏仍可查看日志与按钮。
- `node --check dashboard/server.js` 和 `node --check dashboard/public/app.js` 通过。
